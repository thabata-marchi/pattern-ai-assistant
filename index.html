O que é o GoF?

Gang of Four · 1994
📖 Contexto histórico

GoF = Gang of Four — "A Gangue dos Quatro". São quatro engenheiros que em 1994 publicaram o livro "Design Patterns: Elements of Reusable Object-Oriented Software", a bíblia dos design patterns. Os autores são Erich Gamma, Richard Helm, Ralph Johnson e John Vlissides.

Eles resolveram um problema real: todo mundo estava reinventando a roda. O GoF nomeou e documentou 23 soluções que já funcionavam, criando uma linguagem comum entre devs. Hoje, quando você fala "vou usar um Observer aqui", qualquer dev sênior do mundo entende imediatamente.

📚 Fonte: Gamma, E., Helm, R., Johnson, R., Vlissides, J. (1994). Design Patterns: Elements of Reusable Object-Oriented Software. Addison-Wesley. ISBN 0-201-63361-2.

01

Abstract Factory

Fábrica de fábricas · Família de objetos relacionados
Criação
☕ Dia a Dia

Uma rede de restaurantes no Brasil e no Japão. Cada unidade tem seu cardápio local — mas todas seguem o mesmo manual: entrada, prato principal, sobremesa. A Abstract Factory é esse manual: define a família, cada unidade implementa do seu jeito.

🎯 Quando Usar

Quando precisa criar famílias de objetos relacionados que devem ser usados juntos sem especificar classes concretas. Ex: temas Dark/Light onde botão, input e card mudam juntos.

🐦 Flutter / Dart
// Abstract factory interface abstract class DataSourceFactory { UserDataSource createUserDataSource(); ProductDataSource createProductDataSource(); } // Remote factory class RemoteDataSourceFactory implements DataSourceFactory { @override UserDataSource createUserDataSource() => RemoteUserDataSource(); @override ProductDataSource createProductDataSource() => RemoteProductDataSource(); } // Local factory (offline/mock) class LocalDataSourceFactory implements DataSourceFactory { @override UserDataSource createUserDataSource() => LocalUserDataSource(); @override ProductDataSource createProductDataSource() => LocalProductDataSource(); } // Swap the factory → everything changes together final factory = isOffline ? LocalDataSourceFactory() : RemoteDataSourceFactory();
📓 Caderno

Abstract Factory = Fábrica de famílias. Cria grupos de objetos relacionados sem dizer qual classe exata usar. Troca tudo de uma vez.

família de objetosfábrica de fábricastroca em blocotema/ambiente

02

Factory Method

Delega criação para subclasses · Um tipo por vez
Criação
☕ Dia a Dia

Uma loja de celulares: processo igual para todos — embalagem, entrega, ativação. Mas a Apple faz iPhone, Samsung faz Galaxy. Cada fabricante decide como criar — o processo geral é o mesmo.

🎯 Quando Usar

Quando uma classe não sabe de antemão qual tipo de objeto criar, ou quer delegar a criação para subclasses. Mais simples que Abstract Factory — cria apenas um tipo.

🐦 Flutter / Dart
abstract class Notification { void send(String msg); } class PushNotification implements Notification { @override void send(String msg) => print('PUSH: $msg'); } abstract class NotificationSender { Notification createNotification(); // ← factory method void notify(String msg) => createNotification().send(msg); } class PushSender extends NotificationSender { @override Notification createNotification() => PushNotification(); }
📓 Caderno

Factory Method = Método que cria. Superclasse define "preciso criar algo", subclasse decide o quê.

03

Singleton

Instância única global · Um só objeto
Criação
☕ Dia a Dia

O presidente de um país — em dado momento existe apenas um. Qualquer cidadão que queira falar, fala com a mesma pessoa. Ou a impressora do escritório — todos acessam a mesma.

🎯 Quando Usar

Quando precisa de exatamente uma instância: configurações globais, conexão com banco, log, cache. ⚠️ Cuidado: dificulta testes se abusado.

🐦 Flutter / Dart
class AppConfig { static final AppConfig _instance = AppConfig._internal(); AppConfig._internal(); // private constructor factory AppConfig() => _instance; String apiUrl = 'https://api.montebravo.com'; } final a = AppConfig(); final b = AppConfig(); print(identical(a, b)); // true

Singleton = Um só. Garante uma única instância. Todo mundo acessa o mesmo objeto.

Prototype

Criação
☕ Dia a Dia

A ovelha Dolly — clonada em vez de criada do zero. Ou Ctrl+C, Ctrl+V em um documento complexo: você copia e ajusta, sem recriar tudo.

🎯 Quando Usar

Quando criar do zero é caro e você tem um objeto parecido na memória. Clone e ajuste. Muito usado em BLoC com copyWith.

UserProfile copyWith({String? name, String? email, String? avatarUrl}) => UserProfile(name: name ?? this.name, email: email ?? this.email, avatarUrl: avatarUrl ?? this.avatarUrl); } // In BLoC: newState = currentState.copyWith(name: 'Thabata');

05

Builder

Construção passo a passo · Objetos complexos
Criação
☕ Dia a Dia

Montar um hambúrguer no iFood: pão ✓, queijo ✓, bacon ✗, molho especial ✓. Você adiciona as partes que quer, na ordem certa. Builder monta objetos complexos passo a passo.

🎯 Quando Usar

Quando um objeto tem muitos parâmetros opcionais e o construtor virou um monstro de 15 argumentos. Permite criar versões diferentes com o mesmo processo.

🐦 Flutter / Dart
class ApiRequestBuilder { late String _url; String _method = 'GET'; Map<String,String>? _headers; Map<String,dynamic>? _body; ApiRequestBuilder url(String u) { _url=u; return this; } ApiRequestBuilder post() { _method='POST'; return this; } ApiRequestBuilder withAuth(String t) { _headers={'Authorization':'Bearer $t'}; return this; } ApiRequestBuilder body(Map<String,dynamic> b) { _body=b; return this; } ApiRequest build() => ApiRequest._(_url,_method,_headers,_body); } // Fluent usage: final req = ApiRequestBuilder().url('/suitability').post().withAuth(token).build();
📓 Caderno

Builder = Construção passo a passo. Fluent API. Resolve o construtor monstro. Cada etapa retorna o próprio builder. Termina com .build().

Adapter

Tradutor de interfaces · Compatibiliza o incompatível
Estrutura
☕ Dia a Dia

O adaptador de tomada de viagem. Plugue americano + tomada europeia = não encaixa. O adaptador faz a tradução sem modificar nenhum dos dois.

07

Bridge

Separa abstração da implementação · Duas hierarquias
Estrutura
☕ Dia a Dia

Controle remoto + TV. O controle (abstração) pode ser básico ou avançado. A TV (implementação) pode ser Samsung ou LG. Qualquer controle funciona com qualquer TV via interface comum.

🎯 Quando Usar

Quando quer evitar explosão de subclasses. N formas × M cores = N×M classes sem Bridge, N+M com Bridge. Composição > herança.

🐦 Flutter / Dart
abstract class NotifChannel { void send(String t, String b); } class FCMChannel implements NotifChannel { @override void send(String t, String b) => print('FCM: $t'); } abstract class Notif { final NotifChannel channel; // ← the bridge Notif(this.channel); void notify(); } class AlertNotif extends Notif { AlertNotif(super.channel); @override void notify() => channel.send('Alerta', 'Ação necessária'); }
📓 Caderno

Bridge = Ponte. Separa "o quê" do "como". Evita explosão de subclasses. Composição > herança.

Estrutura hierárquica em árvore onde nós simples e compostos são tratados igual. Menus aninhados, sistema de arquivos, widgets Flutter!

Estrutura

Facade = Fachada. Interface simples para complexidade interna. Services do Clean Architecture são Facades.

🎯 Quando Usar

Adicionar cache, log, lazy loading, controle de acesso sem modificar o objeto real. CachedNetworkImage do Flutter usa este padrão!

🐦 Flutter / Dart

Template Method

Esqueleto fixo com partes customizáveis
